# Claude Code 主 agent 与内置 agent 实现方式（Copilot 结论）

> **来源**：GitHub Copilot 对「主 agent 和内置 agent 是根据提示词模板自动生成，还是写在代码中实现」的总结。

---

## 核心答案

**主要是写在代码/配置中，部分支持自动参数拼装，而非完全自动生成。**

- ❌ 不是完全自动生成  
- ✅ 是写在代码中，辅以动态参数拼装  

当前工程实践更重视**可靠性和可控性**，而非完全的自动化，带来的好处包括：便于版本控制和维护、可审计和可解释、性能和成本可预测。

---

## 详细说明

### 1. 方式一：代码写死（主要方式）

- Anthropic 的 Claude agent 体系中，各个子 agent 的**核心提示词模板**基本写在**代码和配置文件**里。
- 例如：修复代码 agent、搜索 agent 等都有标准化的 prompt 骨架。
- **优点**：可靠、易于版本管理和审计。

### 2. 方式二：动态参数拼装（部分使用）

主 agent 在运行时根据：

- 用户意图  
- 上下文信息  
- 输入内容类型  
- 工具类型  
- …  

自动**选择和填充**合适的提示词模板。  
例如：根据用户问题自动选择「代码解释模板」还是「代码修复模板」。

### 3. 方式三：配置 + 模板引擎

- 采用 **YAML/JSON 配置 + 模板渲染**（如 Jinja2）方式。
- 允许灵活定制不同场景下的 prompt 内容。

### 4. 方式四：元提示词驱动（实验阶段）

- 部分高级场景下，Anthropic 会让模型自己辅助优化/补全 prompt 内容。
- 这是在高度工程化基座上的「微调」，**不是**完全自动生成。

## 与 claude-agent-design.md 的对照

| 维度 | Copilot 结论（本文） | design.md（Google 总结 + 核实） |
|------|----------------------|----------------------------------|
| 是否「完全自动生成」 | ❌ 否，主要写死在代码/配置 | ✅ 一致：执行流程在代码，身份由提示词定义 |
| 提示词存放形态 | 代码 + 配置，部分模板渲染 | 内置部分在 **minified JS** 内；自定义代理用磁盘文件（如 `~/.claude/agents/`） |
| 动态性 | 运行时按意图/上下文/工具**选择与拼装**模板 | 运行时从打包字符串中**选取、拼接、插值**生成 system prompt |
| 工程取向 | 可靠性、可控性、可审计 | 与「代码框架 + 动态提示词注入」一致 |

**综合**：两处结论一致——主 agent 与内置子 agent 的提示词是**写在代码/打包产物中**，通过运行时**动态选择与参数拼装**注入，而非从外部模板文件「自动生成」；自定义代理才使用磁盘上的配置/模板文件。
